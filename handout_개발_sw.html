Claude를 더 잘 쓰는 법 (SW/QA)

입문 핸즈온이 아니라, 2025–2026 최신 베스트프랙티스 + 측정된 근거 기반의 고급 활용·팀 도입 가이드입니다.

① 성과를 가르는 고급 실천

검증 루프를 '코드'가 닫게

테스트·빌드·린터·스크린샷 등 에이전트가 직접 읽는 pass/fail 신호를 주면 사람이 검증 루프가 되지 않고 자율 반복이 됨. (Anthropic 공식 1순위 원칙)

Explore→Plan→Implement→Commit

다중 파일·불확실 작업은 Plan Mode로 탐색·계획 분리. 한 문장으로 diff 설명되는 작은 변경엔 생략.

컨텍스트를 최우선 자원으로

차면 성능 저하(context rot). 작업 전환 시 /clear, 같은 버그를 2번 이상 못 고치면 누적 실패 맥락 버리고 재시작.

서브에이전트로 격리

코드베이스 조사·리뷰는 독립 컨텍스트 서브에이전트에 위임 → 메인은 구현에 집중.

적대적 리뷰 단계

종료 전 신선한 컨텍스트가 diff만 보고 계획 대비 누락 점검(/code-review). "정확성·요구사항만" 지적하게 해 과잉설계 방지.

CLAUDE.md는 짧게·가지치기

"이 줄 빼면 실수하나?"로 판단. 비대해지면 무시됨. 가끔 쓰는 도메인 지식은 Skills로 온디맨드 로드.

결정론은 Hooks로

권고인 CLAUDE.md와 달리 hooks는 매번 강제(예: edit 후 eslint, 특정 폴더 쓰기 차단).

MCP + CLI 연결

이슈트래커·DB·Figma는 MCP로, GitHub·AWS는 gh/aws CLI로(토큰 효율↑).

병렬 세션 + git worktrees

세션별 독립 체크아웃으로 충돌 없이 병렬. 실무 권장 2–4개(이상은 리뷰 부담·rate limit).

대규모 마이그레이션 fan-out

파일 목록 생성 → claude -p 파일별 루프 + --allowedTools로 권한 스코핑. 2–3개로 다듬은 뒤 전량.

Spec-driven

Claude가 먼저 인터뷰→SPEC.md→새 세션(클린 컨텍스트)에서 실행. 자족적 spec(파일/인터페이스/범위외/E2E 검증)이 핵심.

Headless/CI 통합

claude -p ... --output-format json으로 CI·pre-commit·파이프라인에 삽입.

플러그인은 스코프로 배포

커맨드·스킬·hooks·MCP를 한 패키지로. /plugin install <name> --scope project로 팀 표준은 .claude/settings.json에 커밋, 개인 실험은 --scope local. 우선순위 Managed > Local > Project > User — 조직 정책은 개인이 못 끄고, Local로 팀 설정을 나만 오버라이드 가능.

③ 주의 · 한계

· 친숙한 대형/복잡 레포(오래된 임베디드·펌웨어 포함)에선 단순 도입이 느려질 수 있음 → eval로 검증하며 확장.

· 생산성 향상은 종종 "작성→리뷰"로 부담 이동. 적대적 리뷰·테스트 게이트 없으면 기술부채↑.

· 보안은 자동으로 좋아지지 않음. CI 게이트 + 보안 서브에이전트 필수.

· 신뢰 ≠ 자율 허용. 권한 allowlist·샌드박스·hooks·eval로 통제.

④ 멀티벤더 전략 (Claude + Codex + Gemini)

Claude Code를 1차 표준으로 두고, OpenAI Codex(장기 자율·이중화)·Google Gemini(초대형 컨텍스트·멀티모달, 파일럿)를 태스크별로 라우팅하는 전략을 검토할 수 있습니다. 단, 소스코드가 3개 벤더로 송신되므로 데이터·거버넌스 설계가 전제입니다. 자세한 비교·운영 권장안은 AI 에이전트 지형도 문서의 "멀티벤더" 섹션 참고.
