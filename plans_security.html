Claude 요금제 & 보안 비교

"실무에 쓰려면 어떤 플랜이 필요한가"를 보안 관점에서 정리했습니다. 이 문서를 보고 플랜을 정한 뒤, 세미나 전에 계정을 미리 발급해 두면 첫 시간부터 실습이 가능합니다.

결론 먼저 — 권장: Team 이상

업무에 실제로 쓰려면 Team 플랜 이상을 권장합니다. 가장 큰 이유는 보안입니다 — 비즈니스 플랜(Team·Enterprise)은 기본적으로 입력 데이터를 모델 학습에 사용하지 않으며, 중앙 관리·청구·협업(Projects 공유)이 됩니다. 규모가 크거나 SSO·감사 로그·컴플라이언스가 필요하면 Enterprise를 검토하세요.

비용 대비 사용량 — Team vs Enterprise Team·Enterprise 모두 좌석(구독) + 사용량 한도 방식이지 "토큰을 더 사는" 종량제가 아닙니다. 가벼운~중간 사용의 소수 인원이면 좌석 단가가 낮은 Team이 가성비가 좋고, 사용량이 매우 많거나 큰 컨텍스트·높은 한도가 필요하면 Enterprise가 한도·협의 면에서 유리(단가는 높음)합니다. 순수 "토큰당 비용" 최적화가 목적이면 종량제인 API가 가장 정밀합니다. 실제 한도·단가는 영업으로 확인하세요.

보안 관점 — 무엇이 다른가

개인 플랜 (Free / Pro / Max)

· 기본적으로 개인 계정 → 회사가 통제 불가

· 데이터 학습 사용 설정을 개인이 관리(옵트아웃 가능하나 누락 위험)

· 중앙 청구·계정 회수·감사 불가 (퇴사자 계정 통제 어려움)

· 빠른 개인 실험엔 OK, 민감 업무 데이터엔 부적합

비즈니스 플랜 (Team / Enterprise)

· 입력 데이터 기본적으로 모델 학습에 미사용

· 관리자가 좌석 발급/회수·접근 관리 (중앙 통제)

· Projects 팀 공유로 표준·지침 일원화

· Enterprise는 SSO·SCIM·감사 로그·역할 권한·데이터 보존 정책·컴플라이언스(BAA 등) 추가

공통전송·저장 구간 암호화 등 기본 보안은 모든 플랜에 적용됩니다. 그러나 "학습 미사용·중앙 통제·감사"가 조직 도입의 실질적 분기점입니다.
플랜과 무관하게 지켜야 할 원칙어떤 플랜이든 고객 개인정보·미공개 재무·계약/법무 비밀·인증키는 신중히 다루고, 회사 보안팀 가이드를 우선합니다. 플랜은 위험을 줄여줄 뿐, 0으로 만들지 않습니다.

SSO · SCIM 이란? (주로 Enterprise)

SSO — 싱글 사인온

회사 인증 시스템(Okta·Microsoft Entra ID·Google Workspace 등)으로 한 번 로그인하면 Claude도 자동 인증(SAML 2.0). 별도 비밀번호 불필요.

· 회사 보안정책(MFA 등)이 그대로 적용
· 비밀번호 분실·재사용 위험 ↓
· 퇴사·차단 시 중앙에서 막으면 Claude 접근도 차단

SCIM — 계정 자동 동기화

회사 ID 시스템의 사용자 변경을 Claude 좌석에 자동 반영(생성·수정·비활성화).

· 입사 → 자동 좌석 부여
· 부서이동 → 권한/그룹 자동 변경
· 퇴사 → 좌석 자동 회수(수동 누락·보안 구멍 방지)

왜 중요한가인원이 많아질수록 "수동 초대·삭제"는 실수와 보안 구멍을 만듭니다. SSO·SCIM은 접근 통제와 감사를 자동화합니다 — 그래서 전사·규제 환경에선 Enterprise를 권장합니다. (Team은 수동/도메인 기반 기본 관리)

NDA · 계약상 비밀 자료를 입력할 때

대원칙NDA로 보호되는 제3자 자료는 법무 검토와 적절한 계약 없이는 입력하지 않습니다. 외부 AI에 보내는 순간 "제3자 제공"이 되어 NDA의 비밀유지 의무 위반이 될 수 있습니다. (Claude Code도 로컬 파일 내용을 처리 위해 전송하므로 동일하게 주의)

권장 조건 — 대부분 Enterprise + 계약

  • 법무 확인: 해당 NDA가 클라우드 AI(서브프로세서) 처리를 허용하는지 / 사전 서면동의가 필요한지
  • 비즈니스 플랜(Team·Enterprise) 사용 — 입력 데이터 모델 학습 기본 미사용
  • DPA(데이터 처리계약) 체결, 필요 시 Zero Data Retention(데이터 무보존) 옵션 협의 — NDA엔 "보존 최소화"가 핵심
  • 접근 통제: 권한자만 접근, 전용 Project로 격리, 감사 로그 활성(SSO/SCIM)
  • 최소화: 꼭 필요한 부분만, 가능하면 익명화·마스킹 후 입력
  • 데이터 소재지(residency) 요구가 있으면 계약으로 확인
정말 민감하면 — 밖으로 안 보내는 게 최선외부 반출이 곤란한 고강도 NDA·기밀 소스코드 등은 클라우드 AI 대신 온프레미스/로컬 LLM을 검토하세요. "데이터를 회사 밖으로 보내지 않는 것"이 가장 확실한 통제입니다.

국내 규제 관점 — 한국 기업이 추가로 볼 것

해외 클라우드 AI 도입 시 글로벌 보안 기준과 별개로 한국 법령상 검토 포인트가 있습니다. 아래는 법률 자문이 아닌 법무·보안 협의용 체크 포인트입니다.

  • 개인정보보호법(PIPA) — 처리위탁/제3자 제공: 고객·임직원 개인정보를 외부 AI에 입력하는 행위는 처리위탁 또는 제3자 제공에 해당할 수 있어 근거(동의·계약·고지)가 필요합니다. 가장 안전한 운영은 "개인정보는 입력 금지(🔴)"를 기본값으로 두는 것 — 데이터 정책 1장 템플릿의 금지 등급과 연계하세요.
  • 국외 이전: 해외 서버에서 처리되는 서비스에 개인정보를 입력하면 개인정보의 국외 이전 요건(고지·동의 등)이 적용될 수 있습니다. 처리 위치·이전 근거를 계약(DPA)으로 확인하세요.
  • 가명·익명처리: 식별자를 제거·가명처리한 데이터 활용이 원칙적으로 안전합니다. "이름·연락처·식별번호 제거 후 입력"을 실무 규칙으로 명문화하세요.
  • 유출 사고 대응: 개인정보 유출을 인지하면 법정 기한 내 개인정보보호위원회 신고·정보주체 통지 의무가 발생할 수 있습니다. 사내 보고 라인(데이터 정책 템플릿 4장)과 연결해 두세요.
  • 영업비밀·산업기술: 미공개 기술자료는 부정경쟁방지법상 영업비밀 관리 요건(비밀관리성)과 충돌하지 않도록, 입력 허용 범위를 보안팀과 사전 합의하세요. 국가핵심기술 해당 여부는 별도 검토가 필요합니다.
  • 담당자 지정: 개인정보 보호책임자(CPO)·법무 검토를 거쳐 정책을 확정하고, 정책 문서에 담당 연락처를 명시하세요.
면책 고지위 내용은 교육 자료이며 법률 자문이 아닙니다. 실제 도입 정책은 반드시 자사 법무·보안 조직 및 필요시 외부 전문가 검토를 거쳐 확정하세요.

결정 후 — 세미나 전 준비 (도입 담당)

  • 플랜 결정: Team / Enterprise 중 선택 (위 시나리오 참고)
  • 관리자(워크스페이스 Owner) 지정
  • 참석자 수만큼 좌석 발급 및 초대 (세미나 시작 전 로그인 확인)
  • 데이터 사용 정책 설정 점검(학습 미사용 확인) · 보존 정책 결정
  • "허용 데이터 / 금지 데이터" 사내 기준 1장 작성 → 1회차 공유 (템플릿 활용)
  • NDA·고객 비밀 등 민감자료 취급 방침 결정(법무 협의 · 필요시 Enterprise+DPA/ZDR)
  • Claude Code 사용 허용 여부·범위 결정 (3회차 실습 대비)
  • (Enterprise) SSO·SCIM·감사 로그 설정은 IT와 사전 협의
진행자 메모참석자에게는 회사가 발급한 계정으로 로그인하도록 안내하세요. 개인 계정 사용은 보안·통제상 권장하지 않습니다. (사전 준비 가이드 연계)
