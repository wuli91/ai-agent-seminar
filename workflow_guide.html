Session 4에서 본 "반복을 자산으로" — 이 문서는 그걸 실제로 만드는 절차입니다. 세미나에서는 "이런 게 가능하다"까지 봤다면, 여기서는 직접 손으로 설정합니다. 터미널에서 claude가 실행되는 상태(사전 준비 가이드 완료)에서 시작하세요.

팀의 "일하는 방식"을 상시 지침 파일로 (10분)

Step 2 · 슬래시 커맨드

잘 나온 프롬프트를 /주간보고 같은 명령으로 (10분)

Step 3 · MCP 맛보기

외부 도구·데이터 연결의 개념과 주의점 (5분)

Step 4 · 권한 다듬기

시작 위치 아래 모든 작업은 내 업무 폴더(예: 바탕화면의 weekly_report 폴더)에서 합니다. 터미널에서 그 폴더로 이동한 뒤 claude를 실행한 상태를 기준으로 설명합니다.

Step 1 — CLAUDE.md: 매번 다시 설명하지 않기

개념 한 줄: 프로젝트 폴더 최상위에 CLAUDE.md 파일을 두면, 그 폴더에서 claude를 실행할 때마다 자동으로 읽히는 상시 지침이 됩니다.

만드는 법 — 두 가지

  • 자동 초안: 폴더에서 claude 실행 후 /init 입력 → 폴더를 둘러보고 CLAUDE.md 초안을 만들어 줍니다. 초안을 내 업무 규칙으로 다듬으세요.
  • 직접 작성: 메모장/텍스트 편집기로 CLAUDE.md 파일을 만들고 아래 예시 중 하나를 복사해 붙여넣은 뒤, 우리 팀에 맞게 고치세요.
대화 중에도 추가 가능 Claude Code 대화 중 #으로 시작하는 메시지를 보내면 그 내용을 지침으로 기억시킬 수 있습니다. 예: # 보고서 날짜는 항상 YYYY-MM-DD 형식으로 써 — 잘 되는 규칙을 발견할 때마다 바로 적립하세요. 개인 공통 규칙은 ~/.claude/CLAUDE.md(전역)에 두면 모든 폴더에서 적용됩니다.

직무별 작성 예시 — 복사해서 시작하세요

(a) 마케팅팀 — 보고서 형식·톤·금지사항

# CLAUDE.md — 마케팅팀 문서 작업 규칙

## 산출물 형식
- 캠페인 리포트: 요약(3줄) → 핵심 지표 표 → 인사이트 → 다음 액션 순서
- 카피 제안은 항상 3안 이상, 각 안에 타깃·채널 명시
- 보고서는 한 페이지 이내, 표 우선

## 톤
- 사내 보고: 정중한 보고체 / SNS 카피: 친근한 구어체
- 과장 표현("업계 최초", "혁명적") 사용 금지

## 금지사항
- 경쟁사 비방 표현 금지
- 수치는 내가 준 원본 데이터만 사용, 지어내지 말 것
- 확실하지 않은 내용은 [확인 필요]로 표시

(b) HR팀 — 개인정보 금지·법령 출처 표기

# CLAUDE.md — HR팀 문서 작업 규칙

## 개인정보 (가장 중요)
- 직원·지원자의 실명, 연락처, 주민번호, 주소는 절대 출력하지 말 것
- 사람을 지칭할 때는 "지원자 A", "직원 B" 같은 익명 표기만 사용
- 내가 실수로 개인정보를 넣으면, 먼저 지적하고 진행을 멈출 것

## 법령·규정 인용
- 법령을 인용할 때는 반드시 조문 번호와 출처를 함께 표기 (예: 근로기준법 제60조)
- 우리 회사 취업규칙과 법령이 다르면, 둘을 비교 표로 보여줄 것
- 법적 판단이 필요한 내용에는 "노무사/법무 검토 필요" 문구를 붙일 것

## 형식
- 사내 공지문: 제목 → 대상 → 내용 → 시행일 → 문의처 순서
- 톤은 정중하고 중립적으로, 추측성 표현 금지

(c) 기획/PM — 주간보고 형식·표 우선

# CLAUDE.md — 기획/PM 문서 작업 규칙

## 주간보고 형식 (고정)
- 순서: 금주 요약(3줄) → 진행 현황 표 → 이슈/리스크 → 차주 계획
- 진행 현황 표 컬럼: 과제 | 담당 | 상태(완료/진행/지연) | 비고
- 지연 항목은 사유와 대응 방안을 반드시 함께 적을 것

## 회의록
- 결정사항 / 액션 아이템(담당·기한) / 미해결 이슈 세 부분으로 정리
- 액션 아이템은 표로, 기한 없는 항목은 [기한 미정]으로 표시

## 공통
- 설명보다 표 우선, 한 페이지 이내
- 날짜는 YYYY-MM-DD, 숫자는 원본과 대조하고 불확실하면 [확인 필요]
- 사실을 지어내지 말 것 (모르면 모른다고)
확인 방법 CLAUDE.md를 저장한 폴더에서 claude를 새로 실행하고 평소처럼 일을 시켜 보세요. 형식·톤 설명 없이도 규칙대로 나오면 성공입니다. 안 지켜지는 규칙은 더 짧고 구체적으로 고치세요.

Step 2 — 슬래시 커맨드: 잘 나온 프롬프트를 명령으로

개념 한 줄: 프로젝트의 .claude/commands/이름.md 파일을 만들면 /이름으로 호출되는 저장된 프롬프트가 됩니다. 파일 본문이 그대로 프롬프트가 되고, $ARGUMENTS 자리에 명령 뒤에 적은 인자가 들어갑니다.

완성 예시 ①: /주간보고

프로젝트 폴더 안에 .claude/commands/ 폴더를 만들고, 그 안에 주간보고.md 파일을 아래 내용으로 저장하세요.

# 파일 위치: .claude/commands/주간보고.md (아래가 파일 내용 전체)

$ARGUMENTS 주간보고를 작성해줘.

이 폴더의 메모·문서 파일에서 해당 주의 업무 내용을 찾아 정리하되:
1. 금주 요약 3줄
2. 진행 현황 표 (과제 | 담당 | 상태 | 비고)
3. 이슈/리스크 — 지연 항목은 사유와 대응 방안 포함
4. 차주 계획

분량은 한 페이지 이내. 자료에 없는 내용은 지어내지 말고
[확인 필요]로 표시해줘. 결과는 weekly_report.md 파일로 저장해줘.

호출은 Claude Code 대화창에서 이렇게 합니다.

/주간보고 5월 3주차

→ $ARGUMENTS 자리에 "5월 3주차"가 들어가서, "5월 3주차 주간보고를 작성해줘. …"라는 전체 프롬프트가 실행됩니다.

완성 예시 ②: /회의록정리

# 파일 위치: .claude/commands/회의록정리.md

다음 회의록 파일을 정리해줘: $ARGUMENTS

정리 형식:
1. 결정사항 — 불릿으로
2. 액션 아이템 — 표로 (할 일 | 담당자 | 기한), 기한 없으면 [기한 미정]
3. 미해결 이슈 — 다음 회의에서 다룰 것

마지막에 "빠뜨린 위험 요소"가 있는지 한 번 더 점검해서 알려줘.
결과는 원본 파일명 뒤에 _정리.md 를 붙여 저장해줘.
/회의록정리 notes/0512_TF회의.md
"승격" 워크플로우 — 커맨드는 이렇게 늘어납니다 ① 평소처럼 채팅으로 일을 시킨다 → ② 결과가 잘 나온 프롬프트를 발견한다 → ③ 그 프롬프트를 .claude/commands/이름.md로 저장한다 (매번 바뀌는 부분만 $ARGUMENTS로 치환) → ④ 다음부터는 /이름 한 줄로 끝. 처음부터 커맨드를 설계하려 하지 말고, 잘 된 것을 승격시키세요. 프로젝트의 .claude/commands/는 폴더를 공유하면 팀원도 같은 명령을 쓸 수 있고, 나만의 전역 커맨드는 ~/.claude/commands/에 두면 모든 폴더에서 호출됩니다.

Step 3 — MCP 맛보기: 외부 도구·데이터 연결

개념 한 줄: MCP는 Claude를 외부 도구·데이터에 연결하는 표준 규격입니다 — 컴퓨터에 주변기기를 꽂는 USB 포트라고 생각하면 됩니다.

대표 사례

· 구글 드라이브 — 드라이브 문서를 읽어 보고서 재료로
· 사내 위키/시스템 — 사내 지식을 검색해 답변에 반영

추가 방법 (개요)

· claude mcp add 명령으로 서버 추가
· 프로젝트 루트의 .mcp.json 파일로 팀과 공유
· 연결해도 권한 승인 흐름은 그대로 적용됩니다

⚠️ MCP는 데이터가 외부로 나가는 통로입니다 MCP 서버를 연결하면 내 대화·데이터가 그 서버로 전달될 수 있습니다. 반드시 회사가 승인한 목록 안의 MCP만 사용하고, 출처가 불분명한 MCP 서버는 절대 연결하지 마세요. 어떤 서버를 써도 되는지는 도입 담당/보안팀에 먼저 확인하는 것이 원칙입니다.

이 단계는 "혼자 설치"보다 개념과 보안 원칙을 아는 것이 목표입니다. 실제 연결은 도입 담당과 함께 진행하세요.

Step 4 — 권한 다듬기: 안전한 명령만 미리 허용

Claude Code는 파일 수정·명령 실행 전에 승인을 묻습니다. 좋은 안전장치지만, 매번 같은 안전한 명령까지 묻는 게 번거롭다면 프로젝트의 .claude/settings.json에 permissions.allow 목록으로 사전 허용할 수 있습니다.

// 파일 위치: .claude/settings.json (예시)
{
  "permissions": {
    "allow": [
      "Read",
      "Bash(ls:*)",
      "Bash(git status)"
    ]
  }
}

위 예시는 "파일 읽기", "폴더 목록 보기(ls)", "git status" 같은 읽기 전용·조회성 동작만 미리 허용한 것입니다.

계속 물어보게 둘 것 삭제(rm 등)·이동·외부 전송류는 사전 허용하지 마세요. 승인 화면이 마지막 안전벨트입니다 — 번거로움을 줄이는 건 "읽기·조회"까지만.

마무리 — 이번 주 안에 할 일

  • 내 업무 폴더에 CLAUDE.md 1개를 만들었다 (/init 또는 위 예시 복사)
  • 잘 나온 프롬프트 하나를 슬래시 커맨드 1개로 승격했다 (.claude/commands/)
  • 커맨드를 한 번 호출해서 결과를 확인했다 (예: /주간보고 6월 2주차)
  • 회사에서 승인된 MCP가 있는지 도입 담당에게 물어봤다
  • (선택) 읽기·조회성 권한만 settings.json에 사전 허용했다

더 깊이 가려면

  • 심화 실습: hr_advanced 개발 가이드 — 규정 검토 워크플로우를 단계별로 확장하는 예제
