에이전트 AI는 어떻게 작동하나

자주 나오는 용어(MCP·Skill·Tool·Agent·Subagent·Rules), 하네스가 LLM을 에이전트로 감싸는 구조, 그리고 LLM이 텍스트 외 데이터(이미지·음성)를 처리하는 법을 그림으로 설명합니다.

목차 1. 핵심 용어 2. 하네스 = LLM을 에이전트로 3. 구성 계층 4. MCP·Skill·Subagent 5. 멀티모달 입출력 6. 패러다임 변천

1. 핵심 용어 한눈에

LLM (거대 언어 모델)
텍스트(멀티모달) 입력으로 다음 토큰을 예측하는 기억·행동 없는 함수.
비유: 손도 기억도 없이 한 문장씩 받아쓰는 천재
Agent (에이전트)
LLM이 도구를 쓰며 피드백을 받아 루프 안에서 다음 행동을 스스로 결정. = LLM + 루프 + 도구 + 메모리
비유: 천재에게 손·메모·"끝까지 반복" 지시를 준 것
Harness (하네스/스캐폴딩)
LLM을 감싸는 프로그램 계층 — 루프·컨텍스트 조립·도구 실행·파싱·메모리·가드레일.
비유: 천재를 일하게 만드는 사무실·비서·결재 시스템
Tool / Function Calling
모델이 JSON 스키마대로 호출(함수명+인자)을 내면 하네스가 실행하고 결과를 되돌려줌.
비유: 양식에 맞춘 심부름 주문서 → 비서가 실행
MCP (Model Context Protocol)
AI와 외부 도구/데이터를 잇는 개방 표준(Anthropic '24.11 → '25.12 Linux Foundation). 클라이언트-서버.
비유: "AI용 USB-C" — 한 번 만들면 어디에나 꽂힘
Skill (스킬)
SKILL.md+스크립트 폴더. 필요할 때만 단계적 로드(점진적 공개).
비유: MCP=도구 접근, Skill=그 도구로 무엇을 어떻게
Subagent (서브에이전트)
자체 컨텍스트·도구 권한을 가진 별도 에이전트에 하위작업 위임 → 요약만 회수.
비유: 전문팀에 조사 맡기고 한 장 요약만 받기
Rules / CLAUDE.md / Memory
세션마다 주입되는 지속 지시·가드레일. 권고(따를 수도 안 따를 수도) ↔ 강제는 Hook.
비유: CLAUDE.md=사내 위키 / Hook=출입 보안게이트
⚠ 정확성"agent = LLM+loop+tools+memory"는 교육용 요약입니다(Anthropic 공식 표현은 "LLM이 프로세스·도구 사용을 동적으로 주도"). CLAUDE.md는 시스템 프롬프트 뒤 메시지로 전달되는 권고이지 강제 설정이 아닙니다.

2. 하네스가 LLM을 에이전트로 만든다

LLM은 한 번 부르면 텍스트 한 덩어리만 뱉는 정지 함수입니다. 하네스는 이걸 while 루프로 감싸고, 매 턴 컨텍스트를 재조립하며, 도구 실행 결과를 다시 모델 입력으로 넣습니다. 이 되먹임이 "예측기"를 "행동하는 에이전트"로 바꿉니다.

▶ 에이전트 루프 (하네스가 돌리는 사이클)

점선처럼 루프 전체에 얹히는 하네스 부가물: 가드레일·권한(Hook·승인), 메모리 읽기/쓰기, 컨텍스트 압축(요약), 로깅/관측 · 구조 출처: Anthropic — Building Effective Agents (2024.12)

D-level 핵심 메시지"성능 병목은 모델이 아니라 하네스" — 같은 LLM이라도 루프·도구·컨텍스트 설계(=하네스)가 실제 업무 성과를 가릅니다. 두뇌는 LLM, 일하는 능력은 하네스입니다.

3. 구성 계층 — 무엇이 무엇을 감싸나

▶ 에이전트 시스템의 4계층 (바깥이 안을 감쌈) + 실제 환경

구조 출처: Anthropic — Building Effective Agents · 계층 구분은 교육용 재구성

4. MCP · Skill · Subagent 자세히

MCP — "AI용 USB-C"

앱마다 도구마다 따로 연동하면 M×N개의 통합이 필요합니다. MCP라는 공통 규격을 쓰면 M+N개로 줄어듭니다(한 번 만든 서버를 모든 앱이 재사용).

▶ 왜 표준인가 — 연동 수가 곱(M×N)에서 합(M+N)으로

출처: modelcontextprotocol.io — Architecture (Anthropic 2024.11 공개 → 2025.12 Linux Foundation 이관)

▶ MCP 구조 (Host → Client → Server)

서버가 노출하는 3가지: Tools(실행) · Resources(데이터) · Prompts(템플릿)

Skill — 점진적 공개(필요할 때만 펼침)

▶ Skill 로딩 3단계 (목차→챕터→부록)
1
메타데이터만 상시 로드 (이름+설명 — "이런 스킬이 있다"만 인지, 토큰 최소)
↓ 필요하다고 판단하면
2
SKILL.md 본문 로드 (구체 절차·지시)
↓ 더 필요하면
3
번들 파일·스크립트 로드 (예제·코드·참고자료)

→ 컨텍스트(=비용)를 아끼면서 전문 노하우를 주입. MCP=도구 접근, Skill=그 도구로 무엇을 어떻게 할지

Subagent — 컨텍스트 격리 위임

왜 쓰나

조사·리뷰처럼 토큰을 많이 먹는 작업을 별도 컨텍스트에 맡겨, 메인 대화가 깨끗하게 유지됩니다.

어떻게

오케스트레이터(부모)가 작업 위임 → 서브에이전트가 자체 컨텍스트·도구 권한으로 수행 → 요약 결과만 부모에 반환.

5. LLM은 텍스트 외 데이터를 어떻게 다루나

핵심 직관: LLM에게 세상은 "벡터(숫자 좌표)의 나열"입니다. 이미지·음성도 같은 벡터 공간으로 번역하면 텍스트와 똑같이 처리됩니다.

① 텍스트 기본 — 토큰과 임베딩

· 토큰화: 텍스트를 단어/조각(토큰)으로 분할 · 임베딩: 각 토큰을 의미를 담은 숫자 벡터로 변환 · 예측: 트랜스포머가 다음 토큰을 한 개씩 확률적으로 생성

② 비텍스트 입력 — "인코더가 벡터로 번역"

🖼 이미지

패치(작은 조각)로 분할 → 비전 인코더(ViT/CLIP)가 벡터로 → 프로젝터가 LLM 토큰과 같은 차원으로 정렬 → "이미지 토큰"을 텍스트 옆에 이어붙임

🔊 음성

파형 → 스펙트로그램(시간×주파수 이미지) → 오디오 인코더(Whisper식)가 벡터로 → 토큰화

🎬 비디오

프레임 샘플링(예: 초당 1장) → 각 프레임을 이미지처럼 인코딩 + 타임스탬프로 시간 순서 유지

📄 문서/PDF

전통식: OCR로 텍스트 추출 / 네이티브식: 페이지를 이미지로 통째 입력해 표·도표·레이아웃까지 직접 이해(최신 추세)

③ 전체 파이프라인 — 모든 입력이 "한 줄의 토큰"으로

▶ 입력 → 인코더 → 공유 토큰 공간 → 트랜스포머 → 출력

왼→오 한 방향. 모든 입력이 가운데서 "한 줄의 토큰"으로 합쳐지는 것이 핵심. · 도식 출처: Raschka — Understanding Multimodal LLMs (2024)

④ 출력 — 모달리티별 (이해와 생성은 방향이 반대)

텍스트

다음 토큰 생성(가장 기본·안정). 캡션·분석·답변.

이미지 생성

확산: 노이즈에서 점진적으로 그림으로 / AR 토큰: 이미지를 토큰처럼 한 개씩 예측. 이해=이미지→벡터, 생성=조건→픽셀

음성 출력

옛 방식: 텍스트→멜→보코더 / 네이티브 음성(GPT-4o식): 중간 텍스트 없이 직접 오디오 생성 → 빠르고(수백 ms) 감정 표현 풍부

구조화·도구

바운딩박스 좌표·세그멘테이션·JSON·함수 호출 등 특수 토큰으로 출력

⚠ 정확성GPT-4o·Gemini·Claude의 정확한 내부 구조는 비공개입니다. 위 "네이티브/옴니" 설명은 공개 발표 + 오픈소스 유사 모델 기반의 일반적 접근이며, 세부(토크나이저·디코더)는 단정하지 않는 것이 안전합니다.

6. 패러다임 변천 — "프롬프트"에서 "시스템"으로

AI를 잘 쓰는 기술의 최적화 단위가 계속 커져 왔습니다: 문장 1개 → 컨텍스트 윈도우 1개 → 세션을 넘는 시스템 전체. 공통 분모는 모델 가중치(재학습)가 아니라 입력·컨텍스트·도구·외부 구조를 설계한다는 것입니다.

▶ 엔지니어링 단위의 확장 (2022 → 2026) — 바깥이 안을 포함

Anthropic: "context engineering은 prompt engineering의 자연스러운 진화" — 프롬프트는 컨텍스트의 부분집합

▶ 타임라인 — 두 번의 전환, 그리고 부채꼴 분화 (노드별 출처 각인)

③ 구간의 합의된 단일 용어는 아직 없음 — "대체"가 아닌 병렬 분화 (출처별 적대적 교차검증, 2026-06 기준)

3단계(2025–2026)는 단일 용어로 수렴하지 않았다 — 부상 중인 후보들

Harness Engineering (가장 유력)

세션은 분절적("이전 기억 없음") → 진행 파일·git·기능 목록 같은 외부 구조로 여러 컨텍스트 윈도우를 관통. Anthropic: "compaction만으론 최상위 모델도 프로덕션급 미달."

Agent Skills

노하우를 점진적 공개 폴더로 패키징(4장 참고). 컨텍스트 절약형 지식 주입 — 명시적 context engineering 기법.

Spec-Driven Development

코딩 먼저가 아니라 명세(spec)가 공유 진실의 원천 → 계획 → 작업 → 구현. GitHub Spec Kit이 대표 구현.

Agent Control / Tool Design

에이전트의 자율성 정도(eagerness)와 도구 사용 방식을 제어 — OpenAI GPT-5 가이드가 공식화. 프롬프팅 담론이 에이전트 제어로 이동.

Agentic Context Engineering (ACE) 학술

컨텍스트를 진화하는 playbook으로 — 전략을 누적·정제·조직 (Stanford 등, ICLR 2026).

Memory · Self-Evolving 학술

메모리를 에이전트 지능의 일급(first-class) 요소로 격상, 자기 피드백으로 도구·컨텍스트를 스스로 수정하는 에이전트 연구.

⚠ 정확성위 3단계 후보들은 모두 2025–2026 신생 개념으로, "prompt → context → X"의 X에 해당하는 합의된 단일 용어는 아직 없습니다. 학술 서베이(arXiv 2507.13334)는 RAG·메모리·도구·멀티에이전트를 context engineering의 하위 구성요소로 포섭하므로, 실제 구도는 "대체"가 아니라 우산 개념 + 병렬 특화 분야입니다. "harness engineering"이라는 합성어도 Anthropic 공식 표현이 아닌 커뮤니티 명칭입니다.
D-level 핵심 메시지방향은 분명합니다 — "좋은 문장을 쓰는 사람"에서 "에이전트가 일하는 환경을 설계하는 사람"으로. 2장의 하네스, 4장의 Skill/MCP가 바로 그 설계 도구입니다.
