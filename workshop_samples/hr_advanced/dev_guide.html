2. 아키텍처: 3가지 구성요소

이 워크플로우는 Claude Code의 세 가지 메커니즘을 조합합니다.

(a) 지식 연결 (Knowledge)

  • 이 실습에서는 new_labor_law.txt를 "외부 법령 지식원"의 시뮬레이션으로 사용합니다.
  • 에이전트는 매 검토 시 이 파일을 먼저 읽어 "최신 법령"을 컨텍스트에 확보합니다.
  • 실전에서는 이 파일을 법령 검색 MCP 서버나 RAG 시스템으로 교체합니다(5장 참조). 워크플로우 구조는 그대로 유지됩니다.

(b) 페르소나 (Persona)

  • CLAUDE.md는 폴더 최상위에 두면 Claude Code가 세션 시작 시 자동으로 로드하는 프로젝트 메모리 파일입니다.
  • 여기에 노무사 페르소나, 보수적 검토 원칙, 출력 규약을 한 번 정의하면 이 폴더에서 실행되는 모든 세션에 자동 적용됩니다.

(c) 워크플로우 패키징 (Skill/Command)

  • .claude/commands/check_rules.md 파일을 만들면 /check_rules라는 슬래시 커맨드가 생깁니다.
  • 다단계 추론(법령 읽기 → 규칙 읽기 → 조항별 비교 → 리포트 저장)을 한 명령으로 묶습니다.
  • $ARGUMENTS 치환으로 검토 대상 파일을 바꿔가며 재사용할 수 있습니다.

(a) CLAUDE.md 작성 — 페르소나와 출력 규약 주입

# 프로젝트: 취업규칙 컴플라이언스 검토

## 페르소나
너는 대한민국 노동법에 정통한 15년 차 전문 노무사다.
꼼꼼하고 보수적인 관점에서 법 위반 여부를 검토한다.
모호한 경우 "위반 아님"이 아니라 "리스크 있음"으로 분류한다.

## 검토 원칙
- 법령 원문(new_labor_law.txt)을 근거 없이 추측하지 말고, 반드시 파일을 읽고 인용한다.
- 취업규칙의 조항 번호와 법령의 조문 번호를 항상 함께 명시한다.
- 문제 없는 조항도 "검토 완료 - 충돌 없음"으로 표기하여 누락이 없음을 보인다.
- 취업규칙에 법령상 필수 내용이 빠져 있으면 "누락 사항"으로 별도 보고한다.

## 출력 형식 (컴플라이언스 리포트 규약)
모든 검토 리포트는 다음 4개 섹션을 포함한다:
1. **위반/위험 조항 요약** — 문제 조항 번호와 한 줄 요약
2. **상세 비교** — 현행 규정 원문 vs 법령 원문 인용 및 차이 분석
3. **리스크 등급** — 낮음 / 보통 / 높음 (조항별로 판정, 판정 사유 포함)
4. **수정 제안** — 법을 준수하면서 운영 부담을 최소화하는 수정 문구 초안

## 금지 사항
- "위법이다/합법이다" 같은 법적 단정 표현을 쓰지 않는다.
  ("위반 소지가 있음", "충돌 가능성이 높음" 등으로 표현)
- 리포트 말미에 반드시 다음 문구를 포함한다:
  "본 리포트는 AI가 생성한 초안이며, 최종 판단은 노무사/변호사 검토를 거쳐야 합니다."

(b) .claude/commands/check_rules.md 작성 — 워크플로우 패키징

hr_advanced 폴더 안에 .claude/commands/ 디렉터리를 만들고 check_rules.md를 작성합니다. 파일명이 곧 명령 이름이 됩니다(check_rules.md → /check_rules).

mkdir -p .claude/commands

[복사해서 사용할 .claude/commands/check_rules.md 전체 내용]

다음 절차로 취업규칙의 노동법 준수 여부를 검토하라.

1. `new_labor_law.txt`(최신 개정 법령)를 읽고 핵심 조문을 파악한다.
2. `$ARGUMENTS`에 지정된 취업규칙 파일을 읽는다.
   (인자가 비어 있으면 기본값으로 `company_rules.md`를 사용한다)
3. 취업규칙을 조항별로 순회하며 법령과 비교한다:
   - 법령과 충돌하거나 위반 소지가 있는 조항을 모두 찾는다.
   - 충돌이 없는 조항도 "충돌 없음"으로 기록한다.
   - 법령상 필요하지만 취업규칙에 없는 내용은 "누락 사항"으로 기록한다.
4. CLAUDE.md의 출력 규약(요약/상세 비교/리스크 등급/수정 제안)에 따라
   리포트를 작성하고 `compliance_report.md` 파일로 저장한다.
5. 저장 후, 발견한 항목 수와 최고 리스크 등급을 한 줄로 요약 보고한다.

$ARGUMENTS는 명령 실행 시 사용자가 입력한 인자로 치환됩니다. 예: /check_rules company_rules.md를 실행하면 본문의 $ARGUMENTS 자리에 company_rules.md가 들어갑니다.

(c) 실행 및 산출물 확인

폴더 구조가 다음과 같은지 확인합니다.

├── .claude/
│   └── commands/
│       └── check_rules.md         ← (b)에서 작성
├── company_rules.md
└── new_labor_law.txt

Claude Code를 재시작한 뒤 실행합니다(CLAUDE.md와 커맨드는 세션 시작 시 로드되므로, 기존 세션이라면 새로 시작).

/check_rules company_rules.md

확인할 것:

  • compliance_report.md 파일이 생성되었는가
  • 리포트가 4개 섹션 규약(요약/상세 비교/리스크 등급/수정 제안)을 따르는가
  • 제2조 3항(52시간 초과)이 높음, 제3조(가산수당)가 보통 수준의 리스크로 잡혔는가
  • 직장 내 괴롭힘 조항 누락이 보고되었는가
  • 말미에 "전문가 검토 필요" 고지 문구가 있는가

Track A에서 3단계 수동 작업이던 것이 명령 한 번 + 산출물 파일로 바뀌었습니다.

5. 실전 확장 — 시뮬레이션을 진짜 시스템으로

(1) 법령 지식원을 MCP로 교체

new_labor_law.txt는 교육용 시뮬레이션입니다. 실전에서는 MCP(Model Context Protocol) 서버를 연결해 에이전트가 실시간으로 법령을 검색하게 합니다.

# MCP 서버 등록 개요 (서버 구현체에 따라 명령은 달라짐)
claude mcp add law-search -- <법령 검색 MCP 서버 실행 명령>

팀 공유가 필요하면 프로젝트 루트의 .mcp.json에 서버 설정을 정의해 저장소에 포함시킬 수 있습니다. 연결 후에는 check_rules.md의 1단계를 "new_labor_law.txt를 읽는다" 대신 "법령 검색 도구로 최신 근로기준법 조문을 조회한다"로 바꾸면 됩니다. 워크플로우의 나머지 구조는 그대로입니다.

(2) 사내 규정 폴더 일괄 검토 (fan-out)

/check_rules가 파일 인자를 받으므로, 규정이 여러 개라면 파일별로 반복 적용할 수 있습니다.

/check_rules rules/취업규칙.md
/check_rules rules/임금규정.md
/check_rules rules/복무규정.md

또는 커맨드 본문을 "rules/ 폴더의 모든 .md 파일을 각각 검토하고 파일별 리포트를 생성하라"로 확장해 한 번에 일괄 검토하게 만들 수도 있습니다.

(3) 정기 실행

법 개정은 수시로 일어나므로, 분기마다 같은 명령을 재실행해 리포트를 갱신하는 운영 루틴을 만듭니다. 워크플로우가 파일로 패키징되어 있으므로 누가 실행해도 같은 규약의 리포트가 나옵니다.

⚠️ MCP 보안 주의 MCP 서버는 에이전트에게 외부 데이터와 도구 접근 권한을 부여합니다. 출처가 불분명한 MCP 서버는 잘못된 법령 정보 주입이나 데이터 유출 통로가 될 수 있으므로, 회사가 검증·승인한 서버만 연결하세요. 특히 사내 규정 문서를 다루는 워크플로우라면 더욱 그렇습니다.

6. 한계와 검증

AI 산출물은 초안입니다 compliance_report.md는 검토의 출발점이지 결론이 아닙니다. 최종 법적 판단은 반드시 노무사/변호사의 검토를 거쳐야 합니다.
new_labor_law.txt는 교육용 가상 법령입니다 실제 근로기준법 원문이 아니므로, 실무 적용 시에는 법제처(law.go.kr) 등 공식 출처의 법령으로 교체해야 합니다.
에이전트의 인용을 검증하세요 리포트가 인용한 법령 조문이 지식원 원문과 일치하는지 확인하는 습관이 자동화 시대의 핵심 역량입니다. 자동화는 "사람의 검토"를 없애는 것이 아니라, 사람이 검토에만 집중하게 만드는 것입니다.
