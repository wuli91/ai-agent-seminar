개발 · SW × Claude — 첫 1주

"코드를 대신 짜주는 도구"이기 전에 코드베이스를 읽고 설명하고 검증해 주는 동료입니다. 읽기 작업부터 시작해 신뢰를 쌓아가세요.

① 바로 쓰는 대표 시나리오 6

1. 낯선 코드베이스 파악 (온보딩)

새 프로젝트·인수인계 코드에서 구조, 핵심 모듈, 데이터 흐름을 설명받기. 며칠 걸리던 온보딩을 반나절로.

2. 버그 원인 추적

에러 로그를 주고 관련 코드를 찾아 원인 후보를 가능성 순으로 받기. "바로 고치지 말고 원인부터"가 요령.

3. 테스트 코드 작성

기존 테스트 컨벤션을 따르는 단위 테스트 생성 + 직접 실행해 통과 확인까지.

4. 작은 리팩토링

긴 함수 분리, 중복 제거, 네이밍 개선 등 범위가 명확한 정리 작업. 테스트가 있는 코드부터.

5. 릴리스 노트 · 커밋 메시지 · 문서화

diff/커밋 이력으로 릴리스 노트 초안, 커밋 메시지, README·주석 정리.

6. 코드리뷰 사전 점검

PR 올리기 전 내 diff를 먼저 리뷰시켜 명백한 실수·누락을 잡고, 리뷰어의 시간을 아끼기.

② 첫 1주 루틴 — 읽기에서 쓰기로

일정할 일
Day 1 · 설치설치 + 로그인 — 사전 준비 가이드의 명령 그대로 (claude --version 확인까지)
Day 1 · 첫 실행본인 프로젝트 복사본 폴더에서 claude 실행 → /init으로 CLAUDE.md 생성 → "이 코드베이스 구조를 설명해줘"
Day 2~3 · 읽기 전용설명·리뷰·테스트 제안 등 코드를 바꾸지 않는 작업만. 답이 코드와 맞는지 직접 대조하며 감 잡기
Day 4~5 · 작은 수정함수 하나, 버그 하나 수준의 수정 → diff를 확인하고 승인하는 습관 들이기

습관 하나만 들인다면: "바꾸기 전에 계획을 보여줘."

수정 요청 앞에 이 한 문장을 붙이면 Claude가 먼저 계획을 제시하고, 당신이 검토 후 진행합니다. 엉뚱한 방향으로 가는 것을 가장 싸게 막는 방법입니다.

⑤ 주의 · 검증

· 생성 코드는 반드시 diff 리뷰 + 테스트 실행 후 머지. "돌아가니까 머지"는 금지.

· 비밀키·인증서가 있는 폴더에서 실행하지 않기 (.env, 키 파일 등은 사본에서 제외).

· 사내 코드 반출 정책 확인 — 회사가 발급한 계정으로만 사용.

· "AI는 자신 있게 틀린다." 설명이 그럴듯해도 기준은 실제 동작 확인(테스트·실행)입니다.
